# Session 2026-04-06: ARIA Scan Bridge

## What Was Built

**ARIA Scan Bridge** — translates ARIA-OS scan catalog entries directly into quotable/schedulable MillForge jobs.

### Files Created
1. `backend/agents/aria_bridge_agent.py` — ARIABridgeAgent class; MATERIAL_MAP (alloy designations → materials), MATERIAL_MRR dict, FEATURE_TIME dict; methods: catalog_to_quote(), catalog_to_order(), bulk_catalog_to_orders(), part_summary()
2. `backend/agents/stl_analyzer.py` — STLAnalyzer; uses trimesh if available, falls back to binary/ASCII STL header parsing; returns bounding_box, volume_mm3, face_count, complexity
3. `backend/routers/aria_scan.py` — FastAPI router, 5 endpoints: /import, /quote, /bulk-import, /complexity-estimate, /stl-analyze
4. `frontend/src/components/ARIAImport.jsx` — 3-tab UI (JSON Catalog, STL Upload, Batch Import); PartSummaryCard, QuoteCard components
5. `tests/test_aria_scan_bridge.py` — 20 tests covering material mapping, complexity clamping, catalog/STL conversion, HTTP endpoints

### Files Modified
- `backend/main.py` — registered aria_scan_router
- `backend/requirements.txt` — added trimesh>=4.0.0
- `frontend/src/App.jsx` — added ARIAImport to AUTH_TABS

## Key Decisions

- `complexity-estimate` is POST (not GET) — TestClient doesn't support GET with JSON body reliably
- STLAnalyzer gracefully falls back if trimesh not installed (header-only parsing still extracts bounding box)
- Kept router as aria_scan.py (not aria_bridge.py) to avoid overwriting existing job submission bridge
- Used `or 0.0` guard for Optional fields instead of default arguments
